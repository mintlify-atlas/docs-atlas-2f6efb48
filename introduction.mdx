`cross` lets you build and test Rust crates for dozens of targets without touching your system installation. It wraps `cargo` with an identical CLI, pulling in a pre-built container image that includes the right cross toolchain and cross-compiled libraries for your target.

## What problem does cross solve?

Setting up a cross compilation toolchain manually requires downloading the right linker, sysroot, and C libraries for each target — a process that varies by OS, differs between targets, and is easy to break. `cross` eliminates that work entirely: you specify a `--target` flag and it handles everything else inside a container.

## How it works

When you run a `cross` command, it:

1. Reads the `--target` flag to determine which container image to use.
2. Pulls the pre-built image from `ghcr.io/cross-rs/` (once, then cached).
3. Mounts your project and Rust toolchain into the container.
4. Runs the requested `cargo` subcommand inside the container using the target's cross compiler and sysroot.

For `cross test`, `cross run`, and `cross bench`, QEMU user-mode emulation is also launched inside the container so the compiled foreign binary can execute on your x86_64 host.

<Note>
  Testing with QEMU runs unit tests **sequentially** because QEMU becomes unstable with multiple threads. Tests that spawn threads may fail or hang.
</Note>

## Requirements

### Rust

`cross` requires a `rustup`-managed Rust installation. The minimum supported Rust version (MSRV) for `cross` itself is **1.85.0**.

Install rustup from [rustup.rs](https://rustup.rs/).

### Container engine

One of the following is required. If both are installed, `cross` defaults to Docker.

| Engine | Minimum version |
|--------|----------------|
| [Docker](https://www.docker.com) | 20.10 (API 1.40) |
| [Podman](https://podman.io) | 3.4.0 |

<Warning>
  On Linux, Docker requires your user to be in the `docker` group, or you must use rootless Docker. See the [Docker post-installation steps](https://docs.docker.com/engine/install/linux-postinstall/#manage-docker-as-a-non-root-user) for details.
</Warning>

Cross testing also requires a Linux kernel with `binfmt_misc` support so QEMU can transparently execute foreign binaries.
