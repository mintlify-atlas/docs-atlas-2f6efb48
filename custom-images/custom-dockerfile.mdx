title: "Custom Dockerfile"
description: "Write a Dockerfile that extends the official cross base images to add dependencies or customize the build environment."
---

When pre-build hooks are not enough — for example, when you need packages that require compilation steps, complex configuration, or dependencies not available in Ubuntu's repositories — you can provide your own `Dockerfile`. `cross` builds and uses that image instead of the default one.

## Base image naming convention

Every official `cross` image follows this naming pattern:

```
ghcr.io/cross-rs/<TARGET>:<VERSION>
```

Where `<TARGET>` is the Rust target triple (e.g. `aarch64-unknown-linux-gnu`) and `<VERSION>` is the version of `cross` (e.g. `0.2.5` or `latest`).

<Tip>
  Always base your custom image on the official `cross` image for your target. The official images include a complete, pre-configured cross-compilation toolchain. Building one from scratch is complex and error-prone.
</Tip>

## Writing a Dockerfile

### Using a fixed base image

The simplest approach is to `FROM` the official image directly:

```dockerfile
FROM ghcr.io/cross-rs/aarch64-unknown-linux-gnu:latest

RUN dpkg --add-architecture $CROSS_DEB_ARCH && \
    apt-get update && \
    apt-get install --assume-yes libssl-dev:$CROSS_DEB_ARCH
```

`$CROSS_DEB_ARCH` is automatically set by `cross` to the Debian architecture that matches your target (see [Automatic target architecture on Debian](/custom-images/overview#automatic-target-architecture-on-debian)).

### Using `CROSS_BASE_IMAGE`

`cross` injects a build argument named `CROSS_BASE_IMAGE` that points to the default image it would use for the target. This lets you write a single `Dockerfile` that works across targets without hard-coding the image name:

```dockerfile
ARG CROSS_BASE_IMAGE
FROM $CROSS_BASE_IMAGE

RUN dpkg --add-architecture $CROSS_DEB_ARCH && \
    apt-get update && \
    apt-get install --assume-yes libssl-dev:$CROSS_DEB_ARCH
```

<Note>
  You must declare `ARG CROSS_BASE_IMAGE` before the `FROM` instruction for Docker to make it available during the build.
</Note>

## Configuring `Cross.toml`

Reference your `Dockerfile` from `Cross.toml` using either the string shorthand or the full table form.

### Per-target: `target.TARGET.dockerfile`

**String form** — set the path to the `Dockerfile` relative to `Cargo.toml`:

```toml
[target.aarch64-unknown-linux-gnu]
dockerfile = "./Dockerfile"
```

**Table form** — control the build context and pass build arguments:

```toml
[target.aarch64-unknown-linux-gnu.dockerfile]
file = "./Dockerfile"          # path to the Dockerfile, relative to Cargo.toml
context = "."                  # build context directory, defaults to "."
build-args = { ARG1 = "foo" } # passed as --build-arg to the image build
```

### For all targets: `build.dockerfile`

`build.dockerfile` sets a default `Dockerfile` for every target that does not have its own `target.TARGET.dockerfile` entry.

**String form:**

```toml
[build]
dockerfile = "./Dockerfile"
```

**Table form:**

```toml
[build.dockerfile]
file = "./Dockerfile"
context = "."
build-args = { ARG1 = "foo" }
```

<Info>
  A `target.TARGET.dockerfile` entry always takes precedence over `build.dockerfile` for that target.
</Info>

## Full examples

<Tabs>
  <Tab title="aarch64 with libssl">
    Install `libssl-dev` for cross-compilation to `aarch64-unknown-linux-gnu`:

    ```dockerfile
    FROM ghcr.io/cross-rs/aarch64-unknown-linux-gnu:latest

    RUN dpkg --add-architecture $CROSS_DEB_ARCH && \
        apt-get update && \
        apt-get install --assume-yes libssl-dev:$CROSS_DEB_ARCH
    ```

    ```toml
    [target.aarch64-unknown-linux-gnu]
    dockerfile = "./Dockerfile"
    ```
  </Tab>
  <Tab title="Dynamic base with CROSS_BASE_IMAGE">
    A `Dockerfile` that works for any target by using the `CROSS_BASE_IMAGE` argument:

    ```dockerfile
    ARG CROSS_BASE_IMAGE
    FROM $CROSS_BASE_IMAGE

    RUN dpkg --add-architecture $CROSS_DEB_ARCH && \
        apt-get update && \
        apt-get install --assume-yes libssl-dev:$CROSS_DEB_ARCH
    ```

    ```toml
    [build.dockerfile]
    file = "./Dockerfile"
    context = "."
    ```
  </Tab>
  <Tab title="Custom build args">
    Pass build-time variables into the `Dockerfile`:

    ```dockerfile
    ARG CROSS_BASE_IMAGE
    FROM $CROSS_BASE_IMAGE

    ARG MY_PACKAGE_VERSION=1.0.0
    RUN apt-get update && \
        apt-get install --assume-yes my-package=${MY_PACKAGE_VERSION}
    ```

    ```toml
    [target.aarch64-unknown-linux-gnu.dockerfile]
    file = "./Dockerfile"
    context = "."
    build-args = { MY_PACKAGE_VERSION = "2.1.0" }
    ```
  </Tab>
</Tabs>

## FreeBSD helper scripts

Cross's official FreeBSD images ship three helper scripts for managing FreeBSD packages inside pre-build hooks or custom Dockerfiles:

<AccordionGroup>
  <Accordion title="freebsd-fetch-best-mirror.sh">
    Probes a list of known FreeBSD package mirrors and prints the URL of the fastest one that responds. Falls back through backup mirrors and SRV DNS records when the default mirror is unreachable.

    ```bash
    FREEBSD_MIRROR=$(/freebsd-fetch-best-mirror.sh)
    ```
  </Accordion>
  <Accordion title="freebsd-setup-packagesite.sh">
    Downloads and extracts the FreeBSD package site index (`packagesite.yaml`) from the best available mirror into `/opt/freebsd-packagesite/`. This index is required before installing packages.

    ```bash
    /freebsd-setup-packagesite.sh
    ```
  </Accordion>
  <Accordion title="freebsd-install-package.sh">
    Looks up a FreeBSD package by name in the local package site index and installs it into the cross-compilation sysroot.

    ```bash
    /freebsd-install-package.sh <package-name>
    ```
  </Accordion>
</AccordionGroup>

Use all three together in a `pre-build` hook or `RUN` step:

```toml
[target.x86_64-unknown-freebsd]
pre-build = ["""
export FREEBSD_MIRROR=$(/freebsd-fetch-best-mirror.sh) &&
/freebsd-setup-packagesite.sh &&
/freebsd-install-package.sh xen-tools
"""]
```

<Warning>
  These scripts are only available inside the official cross FreeBSD images. They are not present in Linux or other target images.
</Warning>
